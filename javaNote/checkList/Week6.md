### 1. final 사용법을 설명할 수 있는가?

  - 필드에 붙이면 상수 필드가 된다. 스태틱에만 붙인가
  - 로컬 변수에 붙이면 상수가 된다.

### 2. 추상클래스의 용도를 설명할 수 있는가?
  - 추상클래스는 직접적인 사용이 아닌 공통된 필드나 메서드를 서브클래스로 전달해주는 것을 목적으로 작성하는 클래스이다.

### 3. 추상메서드의 용도를 설명할 수 있는가?

### 6. 필드나 메서드의 접근 범위를 조정해야 하는 이유와 조정하는 방법을 설명할 수 있는가?

### 1. 캡슐화를 위해 필드나 메서드에 접근을 제어해야 할 상황을 설명할 수 있는가?

### 2. 접근 제어 modifier의 사용법을 설명할 수 있는가?
  - private: 같은 클래스 내에서만 접근 가능
  - default: 아무것도 선언 안하는 것, 같은 클래스+같은 패키지
  - protected: 같은 클래스 + 같은 패키지 or 서브클래스
  
  - public: 모두 접근(공개)

### 3. getter/setter, property를 설명할 수 있는가?
  - 접근 제한된 필드의 값을 꺼내거나 저장할 때 사용한다.
  - property는 getter/setter를 가리키는 용어이다.

### 4. 추상클래스와 추상메서드 활용하는 상황을 설명할 수 있는가?
  - 서브클래스에게 공통 필드나 메서드를 제공해야 할 때 추상클래스를 사용한다.
  - 오버라이딩하여 재정의할 메서드라면 추상메서드로 선언하고, 서브클래스에서 기능구현을 강제할 때 추상메서드를 사용한다.

# 27일차 (2023-12-20,수)
### 1. 인터페이스 문법의 용도를 설명할 수 있는가?
  - 객체 사용 규칙을 정의하는 문법이다.
  - 직접 기능 구현하지 않기 때문에 추상메서드만 존재한다.

### 2. 인터페이스 상속과 구현을 설명할 수 있는가?
  - 인터페이스 상속
    - 기존 규칙에 새 항목 추가할 때 새 규칙에 구현을 강제하고 싶을 때는 인터페이스 상속을 사용한다.
    - default 메서드는 기존 클래스에 영향을 끼치지 않기 때문에 기능 구현을 강제 할 수 없는데 인터페이스 상속은 가능하다.
    - 인터페이스는 다중 상속 가능 

  - 인터페이스 구현
    - 인터페이스에 정의된 메서드들을 클래스에 구현하는 것을 의미한다.
    - 만약 클래스에서 메서드 구현을 하지 않는다면 추상메서드가 되기 때문에 추상클래스로 설정해야 한다. 
    
### 3. 인터페이스의 다중상속이 가능한 경우와 불가능한 경우를 설명할 수 있는가?
  - 메서드 시그니처는 같아도 리턴타입이 다르면 다중상속이 불가능하다. 
  - 다중구현은 메서드 이름만 같고 리턴타입이나 파라미터가 다르면 불가능하다.

### 4. 인터페이스의 default 메서드의 용도를 설명할 수 있는가?
  - default 메서드는 구현된 메서드로 기존 클래스에 영향을 끼치지 않으면서 기능를 추가할 수 있다.

### 5. 인터페이스의 private 메서드 용도를 설명할 수 있는가?
  - 인터페이스 내에서 default들의 공통코드를 두는 메서드로 인터페이스 내에서만 사용한다. 외부에서 호출 불가.

### 6. 인터페이스와 결합하여 추상클래스를 사용할 때 이점을 설명할 수 있는가?

### 7. Object 클래스에 대해 설명할 수 있는가?

### 8. toString(), equals(), hashCode() 메서드를 설명할 수 있는가?
  - toString:
    - 클래스에 대한 간단한 정보를 리턴한다. (패키지명.클래스명@해시코드)
    - 만약 클래스 정보를 보기 쉽게 출력하려면 toString()을 오버라이딩한다.
    - source -> generate toString() 
  - equals(): 
    - 레퍼런스와 파라미터 인스턴스가 같은지 비교하는 메서드
    - 인스턴스 속의 실질적인 값(데이터)을 비교하고 싶다면 equals()를 오버라이딩한다.
    - source -> generate hashCode() & equals()
  - hashCode():
    -  Object에서 상속받은 hashCode()는 인스턴스마다 고유의 4바이트 정수값(Hash값)을 리턴한다.
    - 인스턴스(메모리)가 다르더라도 같은 데이터를 갖는 경우 
    같은 것으로 취급하려면 오버라이딩한다.
    
### 9. Hash 값의 의미와 용도를 설명할 수 있는가?
  - 해시값이란 데이터를 수학 알고리즘에 따라 연산하여 작은 사이즈의 숫자로 변환한 값
  - Hash 알고리즘에는 MD4, MD5, PGP, SHA 등이 있다.
  - 기능이 좋으면 복잡하고 느린 대신 중복(충돌) 확률이 낮고, 낮으면 단순하고 빠른 대신 중복(충돌) 확률이 높다.

### 1. 인스턴스를 복제하는 방법을 구현할 수 있는가?
  - clone()을 재정의
    - public으로 접근 범위 확장
    - super 명령으로 Object 클래스 기능 그대로 사용
    - clone()의 리턴값은 Object이므로 형변환하여 저장
  - cloneable을 구현하게 함으로써 JVM에게 복제 선언.(clone 기능 활성화)

### 2. shallow copy/deep copy를 설명하고 구현할 수 있는가?
  - shallow copy
    - 객체가 포함하는 의존 객체는 복제하지 않는다.
    - clone() 오버라이딩, cloneable 후 복제할 인스턴스에 clone()하여 복제한다.

  - deep copy
    - 객체가 포함하는 의존객체까지 복제하는 것
    - clone()을 오버라이딩할 때 의존 객체도 복제하는 코드를 개발자가 작성해야 한다.
    - 즉 의존객체 clone()에 대해서도 오버라이딩한다.

### 3. String 클래스의 인스턴스 생성 방법 2가지를 설명할 수 있는가?
  - new:
    - new 명령어로 인스턴스를 생성하면 Heap 메모리에 생성된다.
  - 리터럴:
    - String 리터럴로 String 인스턴스를 만들 때는 constant pool에 생성
    - 한번 만들어지면 중복 저장 안하고 cashe된 데이터를 사용한다.

### 4. String의 equals() 메서드의 동작을 설명할 수 있는가?
  - 문자열이 같은 경우 true를 리턴하도록 오버라이딩되어 있다.

### 5. Immutable 객체와 Mutable 객체를 예를 들어 설명할 수 있는가?
  - Immutable
    - 불변객체는 인스턴스 값을 바꿀 수 없는 객체이다.
    - 값을 바꾸기 위해서는 새 인스턴스를 생성하는 방법 밖에 없다.
    - String이 대표적이다.
  - Mutable
    - 가변 객체는 인스턴스 값을 바꿀 수 있는 객체이다.
    - remove, concat 등을 사용하여 변경할 수 있다.
    - StringBuffer와 StringBuilder가 대표적이다.

### 6. StringBuffer와 StringBuilder의 차이점을 설명할 수 있는가?
  - StringBuffer는 멀티스레드 환경에서 사용해도 안전하다. 동기화 처리가 되기 때문이다. 동기화 처리 lock/unlock 과정에서 시간이 소요되어 Stringbuilder보다 느리다.
  - StringBuilder는 동기화 처리가 되지 않아 빠르지만 Thread Safe하지 않아 싱글스레드 환경에서 적합하다.

### 7. Thread Safe
  - 여러 스레드가 동시에 작업하더라도 문제가 발생하지 않도록 조치를 했음을 의미
    - 조치: 여러 스레드가 같은 변수의 값을 동시에 변경하려 할 때 한 번에 한 스레드만이 작업하도록 제한하는 것
  - 여러 스레드가 동시에 진입해서 명령을 실행하더라도 문제가 발생하지 않는 코드 ex) 변수의 값을 조회만 하는 코드
